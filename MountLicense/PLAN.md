# MountLicense Plugin Plan

## 1. Product Positioning

MountLicense is a lightweight Minecraft vehicle-network plugin. Its core idea is
to turn animals and physical vehicles into registered transport assets that can
be parked, protected, located, summoned within a sane range, and optionally
shared. Public station/rental networks are a later extension, not part of the
core MVP.

The plugin should not compete as a heavy horse RPG, pet leveling, or virtual
stable system. Its value is closer to public infrastructure:

- Players keep using real in-world entities instead of menu-only pets.
- Servers can build roads, docks, stables, terminals, and rental yards around
  vehicles, with public station/rental tooling deferred until the ownership
  core is stable.
- The plugin protects the investment without replacing vanilla travel.
- The default behavior stays small, event-driven, and easy to audit.

Working name: `MountLicense`.

## 2. Design Goals

- Support more than horses. Boats, chest boats, minecarts, camels, pigs,
  striders, and configured animal entities should be handled through the same
  registration model when their API capabilities allow it.
- Treat different entities by capability, not by forcing every entity into a
  horse-shaped flow.
- Keep PDC as the durable identity attached to the entity.
- Keep a small index file only for lookup, station assignment, last-known
  location, and recovery.
- Avoid global entity polling.
- Avoid hard dependency on economy, map, model, or region plugins.
- Make the normal player flow item-driven and immersive.
- Keep commands available for admin recovery, debugging, and precise control.
- Never promise absolute recovery for unloaded entities unless an explicit
  chunk-loading mode is enabled.

## 3. Current Ecosystem Opportunity

Existing horse plugins already cover claim, protection, summon, trust, GUI
management, leveling, and virtual storage. MountLicense should differentiate by
being:

- Entity-type aware instead of horse-only.
- Built around traffic networks: stations, docks, parking slots, rental yards,
  and cargo points.
- Lighter than database-backed pet systems.
- More immersive than command-first ownership plugins.
- Safer for server operators because expensive features are opt-in.

## 4. Supported Object Model

Not every Minecraft entity should be treated as a vehicle. The plugin should use
capability tiers.

### Tier A: Native Transport Assets

These are supported by default because they already function as transport in
vanilla or server plugins.

- `HORSE`
- `DONKEY`
- `MULE`
- `SKELETON_HORSE`
- `ZOMBIE_HORSE`
- `CAMEL`
- `PIG`
- `STRIDER`
- `BOAT`
- `CHEST_BOAT`
- `MINECART`
- `CHEST_MINECART`

Notes:

- Horse-like entities support owner checks, inventory checks, and AI parking.
- Boats and minecarts do not have AI. Their parking model must use locked state,
  passenger entry control, damage protection, and optional movement anchoring.
- Pigs and striders require vanilla steering items. Registration should not
  bypass that survival requirement by default.
- Minecarts are rail-bound assets. They fit stations and ownership better than
  free recall.

### Tier B: Managed Rideable Or Cargo Entities

These can be enabled by config when the server wants broader roleplay or cargo
behavior.

- `LLAMA`
- `TRADER_LLAMA`
- Any future rideable entity exposed by the server API.

These may support registration, protection, inventory access rules, station
assignment, and locate. They should not receive movement promises that vanilla
does not support.

### Tier C: Experimental Configured Entities

Server operators may enable a custom profile for any allowed `LivingEntity` or
`Vehicle` type. Each profile declares which features are active:

- `register`
- `protect`
- `mountAccess`
- `inventoryAccess`
- `summon`
- `park`
- `station`
- `rental`

The default config should keep hostile mobs, bosses, projectiles, displays,
items, and players disabled.

## 5. Persistence Strategy

### Entity PDC: Authority

Every registered vehicle receives persistent data directly on the entity.

Recommended keys:

- `mountlicense_registered`: byte, `1` means this entity is registered.
- `mountlicense_vehicle_id`: string UUID generated by the plugin.
- `mountlicense_owner_uuid`: string UUID of the owner.
- `mountlicense_profile`: string profile id from config, such as `horse`,
  `boat`, or `minecart`.
- `mountlicense_state`: string enum: `ACTIVE`, `PARKED`, `LOCKED`, `RENTAL`,
  `MISSING`, `RELEASED`.
- `mountlicense_station_id`: optional string station id.
- `mountlicense_key_token`: optional string token used to bind physical key
  items.
- `mountlicense_created_at`: long epoch millis.
- `mountlicense_flags`: string or integer bitset for feature flags.

The PDC record is the source of truth. If the index is lost, loaded entities can
be re-indexed from PDC.

### Lightweight Index: Lookup And Recovery

Use `plugins/MountLicense/vehicles.yml` for the MVP. SQLite can be added later
only if the plugin grows into large networks.

Record shape:

```yaml
schema: 1
vehicles:
  <vehicle-id>:
    entityUuid: "<entity-uuid>"
    entityType: "HORSE"
    ownerUuid: "<player-uuid>"
    displayName: "Angus's Mount"
    profile: "horse"
    state: "PARKED"
    world: "world"
    x: 120.5
    y: 64.0
    z: -42.5
    yaw: 90.0
    pitch: 0.0
    stationId: "spawn-stable"
    lastSeenAt: 1779140000000
```

The index is used for:

- `/ml list`
- `/ml locate`
- station membership
- unloaded-entity recovery hints
- admin repair and reindex

It should not be used to mutate an entity that is not loaded unless the chosen
mode explicitly loads chunks.

## 6. Core Player Items

### License

The license registers an eligible entity or vehicle.

Default item:

- Material: `PAPER`
- CustomModelData: configurable
- PDC: `mountlicense_item = license`

Use flow:

1. Player sneaks and right-clicks an eligible target with a license.
2. Plugin validates entity profile, ownership rules, existing registration,
   region rules, and cooldowns.
3. Plugin writes PDC, consumes one license, updates index, plays feedback, and
   sets the display name if enabled.

### Key

The key controls recall, locate, lock, and unlock actions.

Default item:

- Material: `GOAT_HORN` or `CARROT_ON_A_STICK`
- CustomModelData: configurable
- PDC: `mountlicense_item = key`
- Optional bound token: `mountlicense_key_token`

Key modes:

- Unbound key: acts on nearest owned eligible vehicle.
- Bound key: acts on one vehicle id.
- Admin key: permission-gated inspection and override.

### Station Permit

The station permit is reserved for the optional public station/rental extension.
It is not part of the current MVP command or item surface.

Default item:

- Material: `OAK_SIGN` or `LEAD`
- PDC: `mountlicense_item = station_permit`

Station creation should be admin-only by default, with optional player-owned
station support later.

## 7. Registration Rules

Different entities need different claim rules.

### Tameable Or Horse-Like Entities

- Entity must be tamed when the API supports ownership.
- Owner must be the registering player unless the player has admin bypass.
- Entity must not already be registered.
- Entity must not be inside a protected region that blocks registration.
- Optional: entity must be saddled.

### Boats And Minecarts

Boats and minecarts do not have vanilla owners, so the plugin needs an
anti-theft policy.

Default MVP rule:

- Entity must be empty.
- Entity must be unregistered.
- Player must have permission to register vehicles in the current region.
- Optional placement cache can require that the player placed the boat or
  minecart within the last configurable time window.

Stricter server option:

- Only plugin-issued vehicle items can become registrable boats or minecarts.
- A placed custom boat/minecart carries a short-lived placement token.
- Registration succeeds only when token owner matches the player.

### Public Or Rental Vehicles

Admins can register vehicles as public assets. Public vehicles are owned by the
server and checked out by players through station rules.

## 8. Parking And Locking

### Living Mob Parking

For `Mob` entities:

- Set `mountlicense_state = PARKED`.
- Call `setAI(false)` when safe.
- Store last location in the index.
- Optionally set `setPersistent(true)`.
- Optionally disable pickup/leash/mount by non-authorized players.

Do not use `hasAI()` as the only state source. Other plugins can modify AI.

Landing safety:

- If the entity is not on solid ground after dismount, delay parking briefly.
- If the entity is falling or in fluid, skip AI disable and show a warning.

### Boat And Minecart Parking

For entities without AI:

- Set `mountlicense_state = PARKED`.
- Cancel unauthorized entry, damage, destruction, and inventory access.
- Optional movement anchoring can zero velocity or snap to the parking anchor
  when a parked vehicle moves.
- Movement anchoring must be configurable because vehicle movement events can be
  frequent on busy servers.

### Lock Modes

- `PRIVATE`: owner and trusted players only.
- `TRUSTED`: owner plus trust list.
- `PUBLIC_VIEW`: anyone can see or sit, but not move, damage, or access cargo.
- `PUBLIC_RENTAL`: station checkout controls access.
- `ADMIN_ONLY`: locked for moderation or recovery.

## 9. Summon And Locate

Recall should be useful but honest about Minecraft chunk loading.

### Default Recall

The default key recall searches loaded nearby entities.

- Search radius: configurable, default `100`.
- Source: `player.getNearbyEntities(...)`.
- Filter by PDC and owner/trust/rental state.
- Pick nearest bound or eligible vehicle.
- Teleport to a safe location near the player if profile allows teleport.
- Wake living mobs with `setAI(true)` when appropriate.

### Index-Based Locate

If the entity is not loaded:

- Do not silently fail.
- Use the index to show last known world and coordinates.
- Optional compass/glowing effect if the entity is loaded.
- Optional station hint: "last parked at Spawn Stable".

### Optional Chunk-Load Recall

Chunk-load recall is disabled by default.

If enabled:

- Respect cooldowns.
- Limit maximum distance.
- Use scheduler-safe region access.
- Abort on cross-world or unsafe destination unless explicitly allowed.
- Record evidence in docs because this feature is runtime-sensitive.

## 10. Optional Public Stations And Rentals

Private parking is covered by Phase 2 park/lock plus normal server builds such
as fences, docks, and rail yards. A first-class station system is deferred out
of the core MVP. If it returns, it should serve public infrastructure and
rental workflows rather than replacing private stables.

### Station Types

- `STABLE`: horses, camels, pigs, striders, llamas.
- `DOCK`: boats and chest boats.
- `RAIL_BERTH`: minecarts and chest minecarts.
- `YARD`: mixed storage and rental.
- `CARGO`: cargo-focused mounts and storage vehicles.

### Reserved Station Data

The current `VehicleRecord.stationId` field is a reserved compatibility hook.
Do not treat it as a shipped station feature until station commands and storage
exist.

Use `plugins/MountLicense/stations.yml`.

```yaml
schema: 1
stations:
  spawn-stable:
    name: "Spawn Stable"
    type: "STABLE"
    world: "world"
    x: 0.5
    y: 64.0
    z: 0.5
    radius: 8
    ownerUuid: "server"
    public: true
    maxSlots: 12
    allowedProfiles:
      - horse
      - camel
      - donkey
```

### Station Behaviors

- Park registered vehicles into station slots.
- Check in rental vehicles.
- Recall owned vehicles parked at the station when loaded.
- List station vehicles.
- Mark missing vehicles when their last known chunk unloads.
- Optional map marker integration later.

### What Stations Are Not In MVP

Stations should not perform autonomous pathfinding between points in the MVP.
Automatic animal routing is expensive, fragile, and likely to conflict with
chunks, regions, and terrain. The first version should make human-ridden
transport networks reliable, not self-driving.

## 11. Cargo And Inventory Protection

Cargo support depends on entity capabilities.

Entities with inventory:

- `DONKEY`
- `MULE`
- `LLAMA`
- `TRADER_LLAMA`
- `CHEST_BOAT`
- `CHEST_MINECART`

Rules:

- Owner and trusted players can open inventory.
- Rental cargo behavior is configurable.
- Unauthorized inventory open is cancelled with localized feedback.
- Death or destruction drop policy is configurable.
- Admin bypass requires explicit permission.

## 12. Protection Rules

Protection should be broad but not magical.

Protected actions:

- Mount or enter.
- Damage.
- Destroy.
- Inventory open.
- Leash.
- Rename.
- Saddle or equipment modification where API allows.
- Passenger manipulation where API allows.
- Rental checkout bypass.

Owner actions:

- Owner can unlock, release, transfer, or rename.
- Destructive release should require confirmation through command or repeated
  interaction, not a single accidental hit.

Cleaner compatibility:

- Set entity persistent when the API supports it.
- Add scoreboard tag `mountlicense_registered`.
- Keep optional custom name enabled for servers using cleaner whitelist rules.
- Document ClearLag or cleaner configuration. Third-party cleaners will not
  automatically honor MountLicense PDC unless configured to do so.

## 15. Configuration Shape

Suggested files:

- `config.yml`: global behavior.
- `vehicle-profiles.yml`: entity capability profiles.
- `messages.yml`: localized player and admin feedback.
- `vehicles.yml`: runtime vehicle index.
- `stations.yml`: runtime station index.

Example `vehicle-profiles.yml`:

```yaml
profiles:
  horse:
    entityTypes: [HORSE, DONKEY, MULE, SKELETON_HORSE, ZOMBIE_HORSE]
    features:
      register: true
      protect: true
      summon: true
      park: true
      inventoryAccess: true
      station: true
    requiresTamedOwner: true
    parking:
      aiOff: true
      delayIfFallingTicks: 20

  boat:
    entityTypes: [BOAT, CHEST_BOAT]
    features:
      register: true
      protect: true
      summon: true
      park: true
      inventoryAccess: true
      station: true
    requiresTamedOwner: false
    registration:
      requireEmpty: true
      requirePlacementToken: false
    parking:
      anchorMovement: false

  minecart:
    entityTypes: [MINECART, CHEST_MINECART]
    features:
      register: true
      protect: true
      summon: false
      park: true
      inventoryAccess: true
      station: true
    parking:
      anchorMovement: true
```

## 16. Architecture

Suggested package root:

`org.cubexmc.mountlicense`

Packages:

- `MountLicensePlugin`: bootstrap, dependency detection, listener and command
  registration.
- `config`: config load, profile parsing, item definitions.
- `model`: `VehicleRecord`, `VehicleProfile`, `VehicleState`,
  `StationRecord`, `AccessAction`, `TrustEntry`.
- `persistence`: YAML stores, dirty tracking, schema versioning, reindex.
- `service`: registry, access control, parking, recall, station, rental,
  protection, item factory.
- `listener`: item interaction, registration, mount/enter/exit, protection,
  inventory, station interaction, lifecycle.
- `command`: command router and subcommands.
- `scheduler`: Bukkit/Paper/Folia-safe scheduler facade.
- `integration`: optional Vault, region, cleaner, and map hooks.
- `util`: safe location, text, sounds, entity capability helpers.

## 17. Event Map

Registration:

- `PlayerInteractEntityEvent`
- `PlayerInteractEvent`
- Optional placement tracking for boats and minecarts.

Mount and entry:

- `EntityMountEvent`
- `EntityDismountEvent`
- `VehicleEnterEvent`
- `VehicleExitEvent`

Protection:

- `EntityDamageEvent`
- `EntityDamageByEntityEvent`
- `VehicleDamageEvent`
- `VehicleDestroyEvent`
- `InventoryOpenEvent`
- `PlayerLeashEntityEvent`
- `EntityUnleashEvent`

Lifecycle:

- `ChunkLoadEvent`
- `ChunkUnloadEvent`
- `EntityDeathEvent`
- `PlayerQuitEvent`
- Server enable, reload, and disable.

Movement anchoring:

- `VehicleMoveEvent`, only when anchoring is enabled and the vehicle is
  registered and parked.

## 18. Scheduler And Threading Rules

- Bukkit entity, world, inventory, and chunk access must run on the correct
  server thread or region scheduler.
- File IO may run async after a snapshot is built.
- Folia support requires a scheduler abstraction from the first implementation
  slice if Folia is a public support claim.
- Chunk-load recall is R3 runtime work and must be implemented after a
  scheduler policy exists.

## 19. Performance Rules

- No global repeating scan of all worlds.
- Nearby search only when a player uses a key or command.
- Index updates happen on registration, parking, recall, station check-in,
  chunk unload, death, destroy, transfer, and release.
- Dirty runtime files save on a debounced timer and shutdown.
- `VehicleMoveEvent` handling must bail out quickly and only anchor profiles
  that explicitly enable it.
- Cooldowns should apply to recall, locate effects, rental checkout, and admin
  repair actions that touch chunks.

## 20. MVP Scope

### Phase 0: Project Alignment

- Rename or document the project identity away from the current Metro profile.
- Decide target server versions and Java bytecode level.
- Decide whether Folia is supported in v1 or deferred.

### Phase 1: Registry And Ownership

- PDC keys.
- License item.
- Register horse-like entities, boats, and chest boats.
- YAML vehicle index.
- `/ml list`, `/ml info`, `/ml admin inspect`.

### Phase 2: Protection And Parking

- Owner-only mount/enter.
- Inventory protection.
- Damage and destroy protection.
- Living mob parking through AI state.
- Boat parking through lock and optional anchor.
- `/ml lock`, `/ml unlock`, `/ml park`, `/ml release`.

### Phase 3: Key Recall And Locate

- Key item.
- Nearby loaded-entity recall.
- Last-known-location locate.
- Bound key support.
- Cooldowns and clear failure messages.

### Phase 4: Stations

Cancelled from the core MVP. Private stables and docks are handled through
normal world builds plus Phase 2 park/lock. Public stations may return as part
of Phase 5b if a server needs shared infrastructure or checkout flows.

### Phase 5: Sharing And Rentals

- Trust list.
- Transfer.
- Public rental vehicles, station checkout, and check-in are optional Phase 5b
  work, not current release commitments.

### Phase 6: Optional Integrations

- Vault cost for licenses, keys, stations, and rental checkout.
- Region plugin hooks.
- Map markers for stations.
- Cleaner compatibility docs.
- SQLite backend if needed.

## 21. UX Rules

- Common flow should use items first, commands second.
- Every blocked action needs a clear reason.
- Use action bar for short state feedback: parked, unlocked, recalled.
- Use chat for recovery details: last-known location, station, admin warnings.
- Use sounds sparingly and consistently.
- Destructive actions require confirmation.
- Vehicle names are not unique; commands should expose a stable short vehicle id
  when ambiguity matters.
- All player-facing text should be localizable.

## 22. Testing And Verification Plan

Risk by implementation area:

- Docs-only plan: R0.
- Registration and protection services: R2.
- PDC persistence, chunk lifecycle, scheduler, and recall: R3.
- Folia support, release packaging, and version support claims: R4 when changed.

Automated tests to add during implementation:

- Profile parsing and feature flags.
- PDC read/write helpers.
- Vehicle index load/save and schema migration.
- Access control decisions for owner, trusted player, public rental, and admin.
- Parking state transitions.
- Recall selection and cooldown rules.
- Station capacity and profile filtering.

Manual checks for first playable build:

- Register a horse, park it, dismount, remount as owner.
- Non-owner cannot mount, damage, leash, or open cargo.
- Register a boat, lock it, and verify non-owner cannot enter or destroy it.
- Use key recall inside the configured radius.
- Try recall after the entity is unloaded and verify last-known-location
  feedback.
- If Phase 5b is enabled in a future build, park at a public stable and dock
  station.
- Restart server and verify PDC and index reconciliation.

## 23. Open Decisions

- Should the first implementation target Paper/Spigot 1.21.x only, or keep the
  existing repository's older Spigot API baseline?
- Should Folia be a v1 support claim or a later compatibility milestone?
- Should boat and minecart registration require plugin-issued placement tokens
  by default?
- Should chunk-load recall exist at all, or should the plugin stay strictly
  loaded-entity only?
- Decision: station tooling is deferred out of the core MVP; if revived, start
  as admin-owned public infrastructure.
- Should economy costs be part of MVP or left as optional integration?
- Should rental vehicles be in v1 or after core ownership is stable?

## 24. Definition Of Done For MVP

- Registered entities carry PDC and can be reindexed after restart.
- Vehicle index survives restart and gracefully handles missing entities.
- Horse-like entities, boats, and version-available chest boats have working registration,
  protection, parking, and locate behavior.
- Player-facing messages are localizable.
- Admins can inspect, repair, and reindex.
- No global polling is required for normal operation.
- Documentation explains summon limits, cleaner compatibility, version-gated
  entity support tiers, and deferred station/rental boundaries.
